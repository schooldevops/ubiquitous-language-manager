# 데이터 품질 점검 결과

## 1. 점검 대상

- 원천 샘플: `docs/dictionary/sample-data/source-data-dictionary-sample.csv`
- 표준 용어 샘플: `docs/dictionary/sample-data/term-master-sample.csv`
- 표현 매핑 샘플: `docs/dictionary/sample-data/term-expression-sample.csv`
- 별칭 샘플: `docs/dictionary/sample-data/term-alias-sample.csv`
- 관계 샘플: `docs/dictionary/sample-data/term-relationship-sample.csv`

## 2. 점검 결과 요약

| 점검 항목 | 결과 | 비고 |
|---|---|---|
| 한글 용어명 중복 | 없음 | 샘플 기준 중복 없음 |
| 영문 약어 중복 | 없음 | 샘플 기준 중복 없음 |
| 물리 타입 누락 | 없음 | 모든 샘플에 물리 타입 존재 |
| 자릿수 누락 | 없음 | 모든 샘플에 자릿수 존재 |
| 소수점 누락 | 없음 | 모든 샘플에 소수점 존재 |
| 관련 시스템 누락 | 없음 | 모든 원천 샘플에 관련 시스템 존재 |
| 폐기 후보 | 1건 | 고객ID |
| 문맥 확인 후보 | 2건 | 회원번호, 사용자식별자 |

## 3. 폐기 후보

| 원천 용어 | 영문 약어 | 처리 방향 | 사유 |
|---|---|---|---|
| 고객ID | CUST_ID | 고객번호의 금지어 또는 Deprecated 후보로 관리 | 기술 ID와 업무 고객번호 혼동 가능 |

## 4. 문맥 확인 후보

| 원천 용어 | 영문 약어 | 처리 방향 | 사유 |
|---|---|---|---|
| 회원번호 | MBR_NO | 고객번호와 별도 도메인인지 확인 필요 | 회원과 고객의 업무 정의가 다를 수 있음 |
| 사용자식별자 | USER_ID | 기술 사용자 ID인지 업무 고객번호인지 확인 필요 | 인증 계정 식별자와 고객 식별자가 충돌 가능 |

## 5. MVP 테스트에 사용할 핵심 케이스

### 5.1 정확 검색

| 입력 | 기대 결과 |
|---|---|
| 고객번호 | T-000001 고객번호 |
| CUST_NO | T-000001 고객번호 |
| customerNumber | T-000001 고객번호 |
| 주문일자 | T-000005 주문일자 |
| ORD_AMT | T-000007 주문금액 |

### 5.2 유사어 검색

| 입력 | 기대 결과 | 권장 표현 |
|---|---|---|
| 고객ID | 고객번호 | 고객번호 |
| 고객 ID | 고객번호 | 고객번호 |
| customerId | 고객번호 | customerNumber |
| CUST_ID | 고객번호 | CUST_NO |
| 주문 날짜 | 주문일자 | 주문일자 |
| 주문상태 | 주문상태코드 | 주문상태코드 |

### 5.3 기획서 검토

입력:

```text
고객 ID를 입력하면 주문 리스트를 조회한다.
조회 결과에는 주문 날짜, 주문 금액, 주문상태가 표시된다.
```

기대 결과:

```text
고객번호를 입력하면 주문 목록을 조회한다.
조회 결과에는 주문일자, 주문금액, 주문상태코드가 표시된다.
```

## 6. 보완 필요 사항

- 실제 회사 데이터 사전 수신 후 중복 용어와 중복 약어를 재점검해야 한다.
- 관련 시스템은 향후 시스템 마스터와 분리하여 정규화해야 한다.
- `회원번호`, `사용자식별자`는 실제 도메인 정의 확인 후 표준 용어 또는 별칭으로 결정해야 한다.
- `고객ID`는 기술 식별자 의미가 있는 경우 별도 기술 용어로 분리할 수 있다.
