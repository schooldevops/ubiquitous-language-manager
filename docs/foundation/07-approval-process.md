# 용어 승인 프로세스

## 1. 목적

본 문서는 표준 용어, 신규 용어 후보, 용어 수정, 용어 폐기 승인 절차를 정의한다.

## 2. 용어 상태

| 상태 | 의미 | 개발 산출물 사용 |
|---|---|---|
| Draft | 초안 | 불가 |
| Reviewing | 검토 중 | 불가 |
| Approved | 승인 | 가능 |
| Deprecated | 폐기 | 불가 |
| Rejected | 반려 | 불가 |

## 3. 신규 용어 후보 승인 흐름

```text
Draft
  -> Reviewing
  -> Approved
```

반려 시:

```text
Draft 또는 Reviewing
  -> Rejected
```

## 4. 승인 조건

신규 용어가 `Approved` 상태가 되려면 다음 조건을 만족해야 한다.

- 업무 정의가 작성되어 있다.
- 사용 맥락이 작성되어 있다.
- 한글 용어명과 영문 용어명이 작성되어 있다.
- DB 컬럼명 또는 API 필드명 등 사용 예정 표현이 작성되어 있다.
- 기존 유사 용어와 중복 검토가 완료되어 있다.
- 데이터 스튜어드가 의미를 승인했다.
- 개발 리더가 기술 산출물 표현을 검토했다.
- 필요한 경우 아키텍트가 도메인 충돌 여부를 검토했다.

## 5. 폐기 조건

용어를 `Deprecated` 상태로 변경하려면 다음 조건을 만족해야 한다.

- 대체 용어가 지정되어 있다.
- 영향도 분석이 수행되어 있다.
- API, DB, 코드, 문서, 테스트 영향 범위가 정리되어 있다.
- 하위 호환성 유지 또는 마이그레이션 전략이 작성되어 있다.
- 적용 일정이 확정되어 있다.

## 6. 승인 책임

| 대상 | 최종 승인 역할 |
|---|---|
| 신규 용어 | DATA_STEWARD 또는 ADMIN |
| 공통 도메인 용어 | ARCHITECT 또는 ADMIN |
| 용어 폐기 | DATA_STEWARD와 ARCHITECT, 또는 ADMIN |
| API 필드명 변경 | DEVELOPMENT_LEAD와 DATA_STEWARD |
| DB 컬럼명 변경 | DEVELOPMENT_LEAD와 ARCHITECT |

## 7. 감사 기록

승인, 반려, 폐기, 후보 승격은 모두 변경 이력에 기록한다.

필수 기록 항목:

- 대상 용어 ID
- 이전 상태
- 변경 상태
- 요청자
- 승인자
- 변경 사유
- 변경 일시
- 영향도 분석 결과 ID
