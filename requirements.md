# 데이터 사전 기반 유비쿼터스 랭기지 관리 시스템 요구사항

## 1. 개요

본 문서는 기존 데이터 사전을 업무 용어 사전, 유비쿼터스 랭기지, AI 참조 지식 베이스로 확장하여 기획서, API, DB, 코드, 테스트가 동일한 의미 체계를 사용하도록 하는 시스템의 요구사항을 정의한다.

## 2. 목표

- 기획자가 쓰는 용어, 개발자가 쓰는 변수명, DB 컬럼명, API 필드명, 테스트 시나리오, AI가 생성하는 코드가 동일한 표준 의미 체계를 따르도록 한다.
- 기존 데이터 사전의 표준화 범위를 DB 컬럼명, 영문 약어, 물리 타입 관리에서 업무 의미, 산출물 표현, 관계, 영향도, AI 활용 영역까지 확장한다.
- AI가 임의로 용어를 생성하지 않고 사내 표준 용어를 조회, 추천, 검증하도록 한다.
- 신규 용어는 후보, 검토, 승인, 배포 절차를 통해 관리한다.
- RAG와 Graphify를 결합하여 자연어 검색, 관계 검색, 영향도 분석, PR/CI 검증을 지원한다.

## 3. 핵심 개념

### 3.1 데이터 사전 확장 방향

기존 데이터 사전은 다음 수준으로 확장한다.

```text
데이터 사전
  -> 업무 용어 사전
  -> 유비쿼터스 랭기지
  -> AI가 참조하는 표준 지식 베이스
  -> 기획서, API, DB, 코드, 테스트가 연결되는 공통 언어 체계
```

### 3.2 유비쿼터스 랭기지 관리 원칙

유비쿼터스 랭기지는 단순 용어 통일이 아니라 업무 개념, 표현, 기술 산출물, 테스트 용어를 하나의 의미 체계로 연결하는 것을 의미한다.

예를 들어 다음과 같은 표현 혼재는 허용하지 않는다.

| 산출물 | 비표준 표현 |
|---|---|
| 기획서 | 고객 ID |
| API | customerId |
| DB | CUST_NO |
| 코드 | memberNo |
| 화면 | 회원번호 |
| 테스트 | 사용자 식별자 |

표준 관리 기준은 다음과 같다.

| 항목 | 표준 표현 |
|---|---|
| 업무 개념 | 고객번호 |
| 영문명 | Customer Number |
| DB 컬럼명 | CUST_NO |
| API 필드명 | customerNumber |
| 코드 변수명 | customerNumber |
| 화면 표시명 | 고객번호 |
| 테스트 용어 | 고객번호 |
| 기획서 표현 | 고객번호 |

## 5. 데이터 모델 요구사항

### 5.1 용어 마스터

시스템은 표준 용어의 중심 정보를 `TERM_MASTER`로 관리해야 한다.

```text
TERM_MASTER
-----------
TERM_ID
TERM_NO
DOMAIN_NAME
USAGE_TYPE
KOREAN_NAME
ENGLISH_NAME
ENGLISH_ABBREVIATION
BUSINESS_DEFINITION
USAGE_CONTEXT
PHYSICAL_TYPE
DIGITS
DECIMAL_POINT
STATUS
OWNER
VERSION
CREATED_AT
UPDATED_AT
```

예시:

| 항목 | 값 |
|---|---|
| TERM_ID | T-000123 |
| DOMAIN_NAME | 고객 |
| USAGE_TYPE | 표준항목 |
| KOREAN_NAME | 고객번호 |
| ENGLISH_NAME | Customer Number |
| ENGLISH_ABBREVIATION | CUST_NO |
| BUSINESS_DEFINITION | 회사에서 고객을 업무적으로 식별하기 위해 사용하는 번호 |
| USAGE_CONTEXT | 주문, 계약, 청구, 상담 등에서 고객 식별 기준으로 사용 |
| PHYSICAL_TYPE | VARCHAR |
| DIGITS | 20 |
| DECIMAL_POINT | 0 |
| STATUS | Approved |
| OWNER | 고객도메인 데이터스튜어드 |
| VERSION | 1.2 |

### 5.2 용어 표현 매핑

시스템은 기획, API, DB, 코드, UI, 테스트에서 사용하는 표현을 `TERM_EXPRESSION`으로 연결해야 한다.

```text
TERM_EXPRESSION
---------------
TERM_ID
EXPRESSION_TYPE
EXPRESSION_VALUE
LANGUAGE
STYLE
IS_STANDARD
```

예시:

| TERM_ID | EXPRESSION_TYPE | EXPRESSION_VALUE | 설명 |
|---|---|---|---|
| T-000123 | Korean | 고객번호 | 한글 표준명 |
| T-000123 | English | Customer Number | 영문 표준명 |
| T-000123 | DB_COLUMN | CUST_NO | DB 컬럼명 |
| T-000123 | API_FIELD | customerNumber | API 필드명 |
| T-000123 | CODE_VARIABLE | customerNumber | 코드 변수명 |
| T-000123 | UI_LABEL | 고객번호 | 화면 표시명 |
| T-000123 | TEST_WORD | 고객번호 | 테스트 시나리오 용어 |

### 5.3 유사어, 별칭, 금지어

시스템은 유사어, 별칭, 금지어를 `TERM_ALIAS`로 관리해야 한다.

```text
TERM_ALIAS
----------
ALIAS_ID
TERM_ID
ALIAS_NAME
ALIAS_TYPE
RECOMMENDATION_ACTION
REASON
```

예시:

| 별칭 | 유형 | 처리 방식 | 사유 |
|---|---|---|---|
| 고객ID | 유사어 | 고객번호로 변환 권장 | 업무 식별 번호 의미 |
| customerId | 유사어 | customerNumber로 변환 권장 | API 표준 필드명 아님 |
| CUST_ID | 금지어 | CUST_NO 사용 | 기술 ID와 혼동 가능 |
| 회원번호 | 유사어 | 문맥 확인 필요 | 고객과 회원의 도메인 구분 필요 |

### 5.4 용어 관계

시스템은 용어 간 관계를 `TERM_RELATIONSHIP`으로 관리해야 한다.

```text
TERM_RELATIONSHIP
-----------------
SOURCE_TERM_ID
RELATIONSHIP_TYPE
TARGET_TERM_ID
DESCRIPTION
```

지원해야 하는 관계 유형은 다음과 같다.

| 관계 유형 | 의미 |
|---|---|
| sameAs | 같은 의미 |
| synonymOf | 동의어 |
| relatedTo | 관련 용어 |
| partOf | 일부 |
| broaderThan | 상위 개념 |
| narrowerThan | 하위 개념 |
| usedWith | 함께 사용 |
| conflictsWith | 충돌 가능 |
| deprecatedBy | 대체됨 |

### 6.2 용어 검색

시스템은 다음 검색 방식을 모두 제공해야 한다.

#### 6.2.1 정확 검색

다음과 같은 값을 정확히 입력했을 때 해당 표준 용어를 찾아야 한다.

- 고객번호
- CUST_NO
- Customer Number
- customerNumber

#### 6.2.2 유사어 검색

다음과 같은 표현을 입력해도 표준 용어를 추천해야 한다.

- 고객ID
- 고객 식별자
- customer id
- member no
- user number

예시 결과:

| 항목 | 값 |
|---|---|
| 검색어 | 고객ID |
| 추천 표준 용어 | 고객번호 / Customer Number / CUST_NO |
| 설명 | 업무 고객 식별 번호 |
| API 필드 | customerNumber |
| 주의 | CUST_ID는 기술 식별자 의미일 때만 사용 |

#### 6.2.3 의미 기반 검색

시스템은 자연어 문장으로 입력된 의미를 바탕으로 적절한 용어를 찾아야 한다.

예시:

| 검색어 | 추천 용어 | 차이 |
|---|---|---|
| 주문이 발생한 날짜 | 주문일자 / Order Date / ORD_DT | 날짜만 필요할 때 사용 |
| 주문이 발생한 날짜 | 주문일시 / Order Date Time / ORD_DTTM | 시분초까지 필요할 때 사용 |

#### 6.2.4 관계 검색

시스템은 Graphify 기반 관계 검색을 제공해야 한다.

- 고객번호와 함께 자주 쓰이는 용어 조회
- 주문 도메인의 모든 표준 용어 조회
- CUST_NO를 사용하는 시스템 조회
- 폐기된 용어를 아직 사용하는 테이블 조회
- 특정 용어 변경 시 영향받는 API, DB, 코드, 문서 조회

### 6.3 기획서 작성 지원

시스템은 기획서 작성 시 용어를 자동 검토하고 표준 용어를 추천해야 한다.

예시:

```text
입력 문장:
고객 ID를 입력하면 주문 리스트를 조회한다.

검토 결과:
'고객 ID'는 비표준 표현입니다.
표준 용어는 '고객번호'입니다.

수정 권장:
고객번호를 입력하면 주문 목록을 조회한다.
```

기획서 표준화 시 문서 하단에 다음 형식의 용어 매핑표를 생성해야 한다.

| 화면 용어 | 표준 용어 | 영문명 | DB 컬럼 | API 필드 | 설명 |
|---|---|---|---|---|---|
| 고객번호 | 고객번호 | Customer Number | CUST_NO | customerNumber | 고객을 식별하는 업무 번호 |
| 주문번호 | 주문번호 | Order Number | ORD_NO | orderNumber | 주문을 식별하는 번호 |

### 6.4 바이브코딩 지원

AI 개발 도구는 코드, DB, API, DTO, 테스트를 생성할 때 반드시 데이터 사전을 조회해야 한다.

기본 규칙은 다음과 같다.

- 승인된 표준 용어만 개발 산출물에 사용한다.
- 한글 용어, 영문명, 영문 약어, API 필드명, 코드 변수명은 데이터 사전에 등록된 값을 사용한다.
- 데이터 사전에 없는 용어는 임의로 확정하지 않는다.
- 데이터 사전에 없는 용어는 신규 용어 후보로 표시하고 승인 필요 상태로 둔다.
- 폐기된 용어는 사용하지 않고 대체 용어를 안내한다.
- 유사어가 입력되면 표준 용어로 변환한다.

바이브코딩 처리 흐름은 다음과 같다.

```text
사용자 요구사항 입력
  -> AI가 업무 개념 추출
  -> 데이터 사전 검색
  -> 표준 용어 매핑
  -> DB/API/DTO/코드 생성
  -> 표준 위반 여부 검증
  -> 신규 용어 후보 리포트 생성
```

### 6.5 신규 용어 관리

신규 용어는 후보, 검토, 승인, 배포 흐름으로 관리해야 한다.

처리 절차는 다음과 같다.

1. 신규 용어 후보 발생
2. 기존 용어 검색
3. 유사 용어 및 중복 용어 확인
4. 용어 의미 작성
5. 영문명 및 약어 추천
6. 물리 타입, 자릿수, 소수점 제안
7. 관련 시스템 지정
8. 데이터 스튜어드 검토
9. 개발 리더 또는 아키텍트 검토
10. 승인
11. RAG, Graphify, API, IDE 도구에 반영

신규 용어 등록 요청 양식은 다음 항목을 포함해야 한다.

- 요청자
- 요청일
- 관련 프로젝트
- 관련 시스템
- 도메인
- 한글 용어명
- 영문 용어명
- 영문 약어 후보
- 사용 구분
- 물리 타입
- 자릿수
- 소수점
- 업무 정의
- 사용 맥락
- 예시 값
- 기존 유사 용어
- 사용 예정 산출물
- 검토 결과

### 6.6 용어 수정 및 영향도 분석

시스템은 용어 수정 요청 시 수정 유형별 위험도를 구분해야 한다.

| 수정 유형 | 위험도 | 예시 |
|---|---|---|
| 설명 수정 | 낮음 | 업무 정의 보완 |
| 별칭 추가 | 낮음 | 고객ID를 고객번호 별칭으로 추가 |
| API 필드명 수정 | 중간 | customerId -> customerNumber |
| 영문 약어 수정 | 높음 | CUST_ID -> CUST_NO |
| 물리 타입 수정 | 높음 | VARCHAR -> NUMBER |
| 자릿수 수정 | 높음 | 20 -> 30 |
| 용어 폐기 | 높음 | 기존 용어 Deprecated 처리 |

용어 수정 프로세스는 다음과 같다.

1. 수정 요청 접수
2. 영향도 분석
3. 관련 시스템, 테이블, API, 코드 확인
4. 변경안 작성
5. 하위 호환성 검토
6. 데이터 스튜어드 승인
7. 개발팀 승인
8. 적용 일정 확정
9. 변경 버전 배포
10. AI, RAG, Graphify 인덱스 재생성
11. 기존 산출물 마이그레이션

영향도 분석은 다음 대상을 포함해야 한다.

- 시스템
- DB 테이블 및 컬럼
- API 필드
- DTO 및 Entity
- 기획서
- 테스트 케이스
- 연결된 다른 용어

### 6.7 기획-개발 연결

시스템은 기획서의 업무 표현을 데이터 사전을 통해 API, DB, 테스트 산출물로 연결해야 한다.

예시:

| 기획서 표현 | 데이터 사전 매핑 | 산출물 |
|---|---|---|
| 고객번호로 주문 내역을 조회한다 | 고객번호 -> Customer Number -> CUST_NO -> customerNumber | `GET /orders?customerNumber={customerNumber}` |
| 주문 목록에는 주문번호, 주문일자, 주문금액을 표시한다 | 주문번호 -> ORD_NO, 주문일자 -> ORD_DT, 주문금액 -> ORD_AMT | DB 조회 컬럼 |
| 고객번호가 없으면 주문 조회를 할 수 없다 | 고객번호 | 테스트 Given/When/Then |

## 7. AI 프롬프트 요구사항

### 7.1 바이브코딩용 프롬프트 템플릿

시스템은 AI 개발 도구에 다음 규칙을 제공할 수 있어야 한다.

```markdown
# 사내 데이터 사전 기반 개발 규칙

너는 사내 표준 용어를 준수하는 개발 AI다.

## 기본 원칙

1. DB 컬럼명, API 필드명, DTO 변수명, Entity 속성명, 테스트 용어는 반드시 데이터 사전의 표준 용어를 사용한다.
2. 데이터 사전에 없는 용어는 임의로 확정하지 않는다.
3. 없는 용어는 신규 용어 후보로 분리한다.
4. 승인된 용어는 그대로 사용한다.
5. 폐기된 용어는 사용하지 않는다.
6. 유사어가 입력되면 표준 용어로 변환한다.

## 출력 형식

### 1. 업무 개념 추출

| 개념 | 설명 |
|---|---|

### 2. 데이터 사전 매핑

| 개념 | 표준 용어 | 영문명 | DB 컬럼 | API 필드 | 코드 변수 | 상태 |
|---|---|---|---|---|---|---|

### 3. 생성 산출물

- DB DDL
- OpenAPI Schema
- DTO
- Service 코드
- 테스트 코드

### 4. 신규 용어 후보

| 후보 용어 | 추천 영문명 | 추천 약어 | 사유 | 승인 필요 여부 |
|---|---|---|---|---|

### 5. 표준 위반 경고

| 입력 표현 | 문제 | 권장 표현 |
|---|---|---|
```

### 7.2 기획서 작성용 프롬프트 템플릿

```markdown
# 데이터 사전 기반 기획서 작성 규칙

## 작성 규칙

1. 비표준 용어는 표준 용어로 변경한다.
2. 모호한 용어는 후보를 제시한다.
3. 데이터 사전에 없는 용어는 신규 용어 후보로 표시한다.
4. 주요 용어는 문서 하단에 용어 매핑표로 정리한다.
5. API, DB, 화면 표시명으로 이어질 수 있게 작성한다.

## 출력 형식

### 1. 기능 개요
### 2. 사용자 시나리오
### 3. 상세 요구사항
### 4. 예외사항
### 5. 사용 용어

|---|---|---|---|---|---|

### 6. 신규 용어 후보

| 후보 용어 | 의미 | 필요한 이유 | 승인 필요 여부 |

```

## 9. RAG 및 Graphify 연계 요구사항

## 11. 상태 및 거버넌스 요구사항

### 11.1 용어 상태

시스템은 다음 상태를 지원해야 한다.

| 상태 | 기획서 사용 | 개발 사용 | AI 추천 |
|---|---|---|---|
| Draft | 가능 | 불가 | 후보로만 표시 |
| Reviewing | 제한적 가능 | 불가 | 검토 중 표시 |
| Approved | 가능 | 가능 | 우선 추천 |
| Deprecated | 불가 | 불가 | 대체 용어 안내 |
| Rejected | 불가 | 불가 | 추천 금지 |

### 11.2 운영 원칙

- 용어는 개발 단계가 아니라 기획 단계에서 확정해야 한다.
- AI는 신규 용어를 확정하지 않고 후보로만 추천해야 한다.
- 사람은 의미 확정, 표준 승인, 폐기 결정, 도메인 간 충돌 조정을 담당해야 한다.
- Approved 상태의 용어만 개발 산출물에 사용할 수 있어야 한다.
- Deprecated 용어는 신규 산출물에서 사용하지 못하도록 차단해야 한다.

## 15. 비기능 요구사항

- 모든 용어 변경은 이력으로 추적되어야 한다.
- 용어 승인, 폐기, 수정 권한은 역할 기반으로 통제되어야 한다.
- 표준 용어 검색 API는 AI, IDE, PR 봇, CI/CD 파이프라인에서 호출 가능해야 한다.
- RAG 및 Graphify 인덱스는 용어 변경 승인 후 재생성되거나 증분 갱신되어야 한다.
- 표준 위반 검증 결과는 사람이 이해할 수 있는 사유와 권장 표현을 함께 제공해야 한다.
- 기획서, API, DB, 코드, 테스트 간 용어 추적성이 보장되어야 한다.
- 신규 용어 후보는 Approved 상태가 되기 전까지 개발 산출물에 자동 반영되어서는 안 된다.
